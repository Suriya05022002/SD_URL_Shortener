Introduction

Design A URL Shortener

In this chapter, we will tackle an interesting and classic system design interview question: designing a URL shortening service like tinyurl.

Step 1

Step 1 - Understand the problem and establish design scope

System design interview questions are intentionally left open-ended. To design a well-crafted system, it is critical to ask clarification questions.

Interview Dialogue

Candidate: Can you give an example of how a URL shortener work?
Interviewer: Assume URL https://www.systeminterview.com/q=chatsystem&c=loggedin&v=v3&l=long is the original URL. Your service creates an alias with shorter length: https://tinyurl.com/y7keocwj. If you click the alias, it redirects you to the original URL.
Candidate: What is the traffic volume?
Interviewer: 100 million URLs are generated per day.
Candidate: How long is the shortened URL?
Interviewer: As short as possible.
Candidate: What characters are allowed in the shortened URL?
Interviewer: Shortened URL can be a combination of numbers (0-9) and characters (a-z, A-Z).
Candidate: Can shortened URLs be deleted or updated?
Interviewer: For simplicity, let us assume shortened URLs cannot be deleted or updated.

Basic Use Cases

  1. URL shortening: given a long URL => return a much shorter URL
  2. URL redirecting: given a shorter URL => redirect to the original URL
  3. High availability, scalability, and fault tolerance considerations

Traffic Estimation

Metric Calculation Result
Write Operations 100 million URLs / day 100M/day
Write QPS 100M / (24 × 3600) ~1,160 writes/sec
Read:Write Ratio Assumed 10:1 10:1
Read QPS 1,160 × 10 ~11,600 reads/sec

Storage Estimation

Service Lifetime: 10 years Total URLs: 100M × 365 × 10 = 365 billion records Average URL Length: 100 characters Storage Required: 365B × 100 bytes = 36.5 TB

Key Insights

  • Read-heavy system (10:1 read-to-write ratio)
  • Moderate write traffic (~1,160 QPS)
  • High read traffic (~11,600 QPS)
  • Significant storage requirements (36.5 TB over 10 years)

Step 6

Deep Dive: Data Model

Database schema and storage considerations for our URL shortener.

While a hash table is a good starting point, it's not feasible for real-world systems due to memory limitations. We'll use a relational database instead.

Database Schema

Column Type Description
id BIGINT Primary key, auto-increment
shortUrl VARCHAR(7) Short URL hash value
longUrl VARCHAR(2048) Original long URL
createdAt DATETIME Creation timestamp
expiresAt DATETIME Expiration timestamp (optional)

Note: We use VARCHAR(7) for shortUrl because our hash function will generate 7-character strings (explained in the next section).

Database Indexing

  • Primary Index: On `id` column for fast lookups
  • Unique Index: On `shortUrl` column to prevent duplicates
  • Index: On `longUrl` column for reverse lookups
Step 7

Deep Dive: Hash Function

Exploring two approaches: Hash + Collision Resolution vs Base 62 Conversion.

Hash Value Length Calculation

With characters [0-9, a-z, A-Z], we have 62 possible characters. We need to find the smallest n where 62^n ≥ 365 billion URLs:

Length (n) Maximum URLs (62^n) Sufficient?
1 62 ❌
2 3,844 ❌
3 238,328 ❌
4 14,776,336 ❌
5 916,132,832 ❌
6 56,800,235,584 ❌
7 3,521,614,606,208 (~3.5 trillion) ✅

Result

We need 7 characters for our hash value to support 365 billion URLs with room for growth.

Step 8

Hash + Collision Resolution vs Base 62

Exploring two approaches: Hash + Collision Resolution vs Base 62 Conversion.

Approach 1: Hash + Collision Resolution

Use well-known hash functions like CRC32, MD5, or SHA-1, then take the first 7 characters:

Hash Function Hash Value (Hex) First 7 Chars
CRC32 5cb54054 5cb5405
MD5 5a62509a84df9ee03fe1230b9df8b84e 5a62509
SHA-1 0eeae7916c06853901d9ccbefbfcaf4de57ed85b 0eeae79

Collision Resolution

When collisions occur, append a predefined string and rehash until no collision exists:

1. Hash "https://example.com" → "abc1234"
2. Check if "abc1234" exists in database
3. If exists, hash "https://example.com" + "1" → "def5678"
4. Repeat until unique hash found

Approach 2: Base 62 Conversion

Convert a unique ID to base 62 representation using our 62-character set:

Character Set: [0-9, a-z, A-Z] Mapping: 0→0, 1→1, ..., 9→9, 10→a, 11→b, ..., 35→z, 36→A, ..., 61→Z Example: Convert ID 11157 to base 62 11157₁₀ = 2×62² + 55×62¹ + 59×62⁰ = [2, 55, 59] → [2, T, X] Result: "2TX"

Comparison of Approaches

Aspect Hash + Collision Resolution Base 62 Conversion
URL Length Fixed length (7 chars) Variable length (grows with ID)
ID Generator Not required Requires unique ID generator
Collisions Possible, needs resolution Not possible (unique IDs)
Predictability Hard to predict next URL Easy to predict (security concern)
Performance May require multiple DB queries Single operation

Recommendation: We'll use Base 62 conversion for its simplicity and performance benefits.

Step 9

URL Shortening & Redirecting Implementation

Step-by-step process of creating short URLs using Base 62 conversion.

URL Shortening Process
1. Client sends POST request with longURL
2. Check if longURL already exists in database
3. If exists: return existing shortURL
4. If not: generate unique ID
5. Convert ID to shortURL using Base 62
6. Save ID, shortURL, longURL to database
7. Return shortURL to client

Concrete Example

Input: https://en.wikipedia.org/wiki/Systems_design
Unique ID: 2009215674938
Base 62 Conversion: 2009215674938 → "zn9edcu"
Short URL: https://tinyurl.com/zn9edcu
ID Short URL Long URL
2009215674938 zn9edcu https://en.wikipedia.org/wiki/Systems_design

Unique ID Generator

The distributed unique ID generator is crucial for our system. Several approaches exist:

  • Auto-increment Database ID: Simple but not scalable
  • UUID: Globally unique but too long
  • Snowflake Algorithm: Twitter's approach with timestamp + machine ID + sequence
  • Database Ticket Servers: Centralized ID generation

Recommended: Snowflake Algorithm

Generates 64-bit IDs with timestamp, datacenter ID, machine ID, and sequence number for high performance and uniqueness.

URL Redirecting Process
1. User clicks short URL: https://tinyurl.com/zn9edcu 2. Load balancer forwards request to web servers 3. Check cache for shortURL → longURL mapping 4. If in cache: return longURL directly 5. If not in cache: query database 6. If found: cache result and return longURL 7. If not found: return 404 error 8. Browser redirects user to longURL

Performance Optimization

Since reads are 10x more frequent than writes, caching is essential:

  • Cache Hit: Return result immediately (~1ms)
  • Cache Miss: Query database, cache result, return (~10-50ms)
  • Cache Strategy: LRU (Least Recently Used) eviction
  • Cache TTL: 24 hours for popular URLs

Performance Tip: Popular URLs (80/20 rule) should have high cache hit rates, dramatically reducing database load.

FAQs About URL Shortener Design

Common questions about URL shortener architecture and implementation

Q1. Why use Base 62 instead of Base 64 encoding?

+

Base 62 uses only alphanumeric characters [0-9, a-z, A-Z], avoiding URL-unsafe characters like '+' and '/' found in Base 64. This ensures the shortened URLs work reliably across all systems without encoding issues.

Q2. How do you handle URL expiration?

+

Add an 'expiresAt' column to the database. During redirects, check if the current time exceeds the expiration time. Use a background job to periodically clean up expired URLs and free up storage space.

Q3. What if the unique ID generator fails?

+

Implement multiple ID generator instances with different ranges (e.g., Server 1: 1-1M, Server 2: 1M-2M). Use a load balancer with health checks to route requests to healthy instances. Consider using distributed algorithms like Snowflake for better resilience.

Q4. How do you prevent malicious URLs?

+

Implement URL validation against blacklists of known malicious domains, use third-party security services for real-time threat detection, and provide user reporting mechanisms. Consider scanning URLs with antivirus APIs before shortening.

Q5. How do you handle database sharding?

+

Use consistent hashing based on the short URL to distribute data evenly across shards. Implement a routing layer that determines which shard contains the data. Plan for shard rebalancing when adding new database nodes.
